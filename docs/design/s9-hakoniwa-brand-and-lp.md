## 1. ブランド世界観 — HAKO-NIWA（箱庭）

### 1.1 名称・タグライン
| 項目 | 値 | 使い方 |
|------|----|--------|
| 主名称（ロゴ・英字） | **HAKO-NIWA** | ヘッダ・ロゴ・metadata の `title` 先頭・英語文脈 |
| 副名称（日本語） | **箱庭** | 日本語文脈での併記。本文中・説明文・和文見出し |
| タグライン | **みんなが出会える場所** | ヒーロー直下・metadata description・OGP |
| 旧名 | ~~rendez~~ | **全置換・残存ゼロ**（コード / docs / metadata / 規約） |

表記ルール:
- ロゴは `HAKO-NIWA`（大文字・ハイフン）。サブに小さく「箱庭」。
- 本文で初出時は「HAKO-NIWA（箱庭）」、以降は文脈に応じ「箱庭」可。
- タグラインは句点なし・10 文字「みんなが出会える場所」で固定。改変しない。

### 1.2 ブランドの比喩（世界観の核）
> **箱庭** ＝ 小さくて、安心できる、手入れの行き届いた庭。
> そこに人が集い、自然に出会う。

この比喩から導く設計上の含意:
- **「小さい」** = 3 対 3 ＝ 6 人。大勢の中で埋もれない、ちょうどよい人数。誇大に「たくさんの出会い」と言わない。
- **「安心できる」** = 本人確認・運営手配・評価で守られた場。安全を**静かに**伝える（煽らない）。
- **「手入れが行き届いた」** = 運営が枠・会場・質を整える。ユーザーは探し回らなくてよい。
- **「自然に出会う」** = アプリ内チャット無し。作為的なやり取りではなく、場の力で会う。

### 1.3 コンセプトの幅（少しだけ上位へ）
- 旧来の一点突破「**恋愛の合コン**」から、上位概念を **「安心できる場で、人と出会う」** に少し広げる。
- ただし**主機能は不変**: 男女 **3 対 3 ＝ 6 人で会う**を中心に据える。
- 文言は「出会い・つながり」を含む広めのトーンにするが、**「合コン」という語は禁止しない**（検索性・分かりやすさのため、説明文で 1 回程度は補助的に使ってよい。主見出しでは使わない）。
- 誇大表現は引き続き禁止（design-system §8）。「運命」「最高の」「業界 No.1」は不可。

**主見出しで使う語の優先度:**
1. 「出会う / 会いにいく / 集まる」（◎ 主役）
2. 「安心 / ちょうどいい人数 / 6 人で」（◎ 補強）
3. 「3 対 3」（○ 事実として明示。見出しにも可）
4. 「合コン」（△ 説明文での補助のみ。主見出し不可）

### 1.4 ボイス＆トーン（コピーの書き方）
design-system §0「機能的で誠実」を継承し、HAKO-NIWA 用に具体化:
- **静かな敬体**。「〜します」「〜いただけます」。感嘆符は原則使わない。
- **事実から書く**。「6 人で会います」「会場はこちらで手配します」。効能を盛らない。
- **庭の言葉を控えめに 1〜2 箇所だけ**。「整えています」「お迎えします」程度。比喩を全面に塗らない（やりすぎは AI っぽい）。
- 禁止: 「運命の出会い」「今すぐ無料登録」「業界初」「マッチング率○○%」等の煽り（§8）。
- 絵文字は祝意の例外（🎁 初回無料）1 つまで。LP・オンボに絵文字を撒かない。

トーンの可否（早見）:
| ✓ 採用 | ✗ 不採用 |
|--------|----------|
| みんなが出会える場所 | 運命の人に出会おう |
| 3 対 3 で、会いにいく。 | 今すぐ理想の相手を見つけよう！ |
| 安心して会える場を整えています | 業界最安・最速マッチング |
| 会場はこちらで手配します | 出会いは無限大🌟 |

### 1.5 反映先（ブランド名の置換対象 — 設計上の指示）
frontend-worker / 将軍が `rendez` → `HAKO-NIWA` を置換する箇所（本書は文言面の指示）:
- `src/app/layout.tsx` の metadata（title / description / OGP）
- `src/app/LoginScreen.tsx`（ロゴ表記・コピー）→ §3 で再設計
- `src/components/ComingSoon.tsx` / `src/app/coming-soon/page.tsx`（metadata 含む）→ §3 で再設計
- README / docs 全般 / 規約類（`/legal/*`）
- metadata の例: `title: "HAKO-NIWA（箱庭）— みんなが出会える場所"` / `description: "HAKO-NIWA（箱庭）は、男女3人ずつ・計6人で会う、安心できる出会いの場です。東京・恵比寿 / 池袋 / 銀座。"`

## 2. 箱庭モチーフの最小 SVG（画像ゼロでも成立させる装飾）

殿の不満の核心は「**画像が無いまま枠だけが残り、文言が浮いて謎になる**」こと。
対策の原則は **「画像を待たない設計」**: 写真・イラストは*あれば*主役だが、**無くてもタイポと余白と最小 SVG で完結**させる。

### 2.1 方針
- `public/` に**外部依存ゼロ・軽量（各 2KB 未満目安）の SVG** を最小限だけ置く。
- すべて **線画（stroke）**・1.5〜2px・単色（`currentColor` で `ink` / `line` / `accent` を流用）。塗りはほぼ使わない（design-system §4.6）。
- フラットな SaaS イラストや 3D レンダは禁止。**手描き感のある編集的な線**（少し不均一な、製図ではない庭の素描）。
- 役割は**主役ではなく沈黙の調度**。意味を説明しすぎない小さな添景に留める（§8「根拠のない装飾過多」回避）。

### 2.2 用意する SVG（5 点・命名と用途）
すべて `public/brand/` 配下を想定。`viewBox` は正方 or 横長を明記。`fill="none" stroke="currentColor"`。

| ファイル | 内容（線画イメージ） | viewBox | 主な使用箇所 | 色（currentColor を当てる側） |
|----------|---------------------|---------|--------------|------------------------------|
| `mark.svg` | **箱庭マーク**: 角丸の枠（箱）の中に、小さな区画線と植栽 2〜3 本の点描。現行 `◇` の置き換え。 | 0 0 44 44 | ロゴ左のマーク（LoginScreen / ComingSoon / AppHeader 任意） | `line-200` 枠 + `accent-500` 植栽 |
| `garden-plot.svg` | **小さな庭の俯瞰**: 長方形の区画を細線で 3〜4 マスに割り、各マスに丸い植え込み・飛び石。人の比喩として小さな○を 6 つ控えめに配置（3 対 3 ＝ 6 人の暗喩、ただし顔は描かない）。 | 0 0 320 200 | LP ヒーローの添景（写真の代替）/ オンボ各スライドの上部 | `line-200` 主体 + `accent-300` 差し色 1 点 |
| `lantern.svg` | **石灯籠／小さな灯り**の線画。「安心・灯り」の象徴。 | 0 0 64 64 | 「安心」価値ブロックの小アイコン | `ink-500` |
| `gate.svg` | **小さな門（枝折戸）**の線画。「入口・はじまり」。 | 0 0 64 64 | CTA 近傍 / オンボ「はじめる」スライド | `ink-500` |
| `stepping-stones.svg` | **飛び石 3 つ**が奥へ続く線画。「流れ・進み方」。 | 0 0 96 48 | 「ご利用の流れ」セクション見出し脇 | `line-200` + `accent-300` |

> 6 つの○について: **顔・人型は描かない**（等身大の写真主義 §4.6 と矛盾させない・記号的な人アイコンの多用回避）。あくまで「区画に置かれた小石／植栽」の抽象。3 つ・3 つで対になる配置にして、見る人が自然に「6 人 / 3 対 3」を感じる程度に留める。

### 2.3 装飾としての置き方（実装ガイド）
- **サイズ**: ヒーロー添景は幅 100%・最大 320px・中央寄せ。アイコンは 28〜40px。
- **彩度を上げない**: SVG はほぼニュートラル線画。`accent` は 1 SVG につき**最大 1 点**の差し色のみ（植栽の実・飛び石 1 枚など）。
- **背景に沈ませる**: 不透明度 100% でも線が細いので主張しない。必要なら `opacity: 0.9`。
- **`aria-hidden`** を付け、装飾として扱う（情報は隣接テキストが担う）。
- **写真が将来入る場合**: ヒーローの SVG 添景はそのまま「写真が無いときのフォールバック」として機能させる設計にする（写真ありなら写真、無ければ `garden-plot.svg`）。これで「画像欠落で枠だけ残る」事故が構造的に起きない。

### 2.4 現行 `◇` からの移行
現状コードは至る所で `◇`（ダイヤ字）をプレースホルダに使っている（LoginScreen / ComingSoon / EmptyState 等）。
- ロゴ位置の `◇` → `mark.svg`（箱庭マーク）に置換。
- オンボの中央プレースホルダ（現 `aspect-[4/3]` の `◇` 箱）→ §4 のとおり `garden-plot.svg` または編集的レイアウトに置換。
- EmptyState 等の `◇` は**そのまま残してよい**（空状態の中立記号として機能しており、過剰に SVG 化しない）。

## 3. LP（ランディング）の設計

対象 2 画面:
- **`/`（LoginScreen）** — LIFF 内・未ログインの実質トップ。ログインへの主導線。
- **`/coming-soon`（ComingSoon）** — リリース前の待機画面 / 全体ゲートの誘導先。

両者は**同じブランド世界観・同じ縦構成**を共有し、CTA だけが異なる（LoginScreen＝動く / ComingSoon＝準備中）。
ペルソナ: 20 代後半〜30 代後半、東京（恵比寿 / 池袋 / 銀座）、1 対 1 マッチングアプリ疲れで「**安心して会いたい**」層。

### 3.1 LP の情報設計（縦の重みづけ）
スマホ縦・単一カラム。上から:
1. **ブランドマーク + 名称 + エリア**（軽く）
2. **ヒーロー**: 主見出し（明朝）+ タグライン + 1 文の説明 + 箱庭の添景 SVG
3. **価値訴求 5 点**（編集的リスト。3 カラム特徴の量産構図にしない）
4. **ご利用の流れ**（4〜5 ステップを簡潔に）
5. **CTA 2 導線**（固定 or 末尾）
6. **規約リンク**（小さく）

> 量産 LP 回避: 「巨大ヒーロー + 3 カラムカード + キラキラ」を作らない（§8）。価値は**縦に積む編集的リスト**で、各項目に小さな線アイコン + 1 行見出し + 1〜2 行本文。

- 通知希望プレースホルダは**現行の誠実トーンを維持**（動かないものを動くように見せない）。「準備中」非活性のまま。
- ComingSoon は presentational（hooks 無し）の制約を維持 — CTA は静的のまま。

### 3.4 画像が無くても成立する根拠（殿の懸念への回答）
この LP は **写真 0 枚で完結**する。成立の担保:
1. **ヒーローは添景 SVG が常に入る**（写真の有無に依存しない）。枠だけ残る事故が起きない。
2. **価値・流れはテキスト + 線アイコン**で構成し、画像領域を主役にしない。
3. **謎文言の排除**: 「ここに画像」的なプレースホルダ文言を一切置かない。代わりに**意味のある添景 SVG** か、**余白**を置く。
4. 将来写真を入れる場合も、SVG はフォールバックとして残るため、差し替え漏れがあっても破綻しない。

- **女性に ¥2,000 を表示しない**を徹底（一覧カード含む）。女性視点で金額行が出る箇所をゼロにする。

- 一覧カード（`PublicSlotCard`）: `男性 ¥2,000 ・ 女性 無料`（中黒区切り・`tabular-nums`）。
- 詳細（explore）: `男性 ¥2,000（女性は無料です）`（現行維持）。
- いずれも**販促色を付けない**（`ink-700` / `ink-500`、accent で煽らない）。

### 5.4 実装の含意（frontend-worker）
- `SlotCard`（authenticated）に **viewer gender を渡す**改修が必要（現状 gender 非依存で feeMale 固定表示）。`browse/page.tsx` は既に `getMe()` 相当で gender を取得しているか要確認。取得経路があれば card に prop で渡し、女性は料金行を出さない。
- `PublicSlotCard`（public）は gender を持てない（公開 DTO に無い）→ **中立併記**（女性無料を追記）に変更。
- `PaymentNotice` は既に gender 出し分け済み → そのまま流用（U-05 / U-06 / U-14 で再利用）。
- `ApplyConfirmSheet` の「料金について確認しました」チェックは**女性・男性初回には出さない**（金額確認の対象外）。

## 6. 撮影後フローの導線設計（いつ何が見え、いつ予約できるか）

殿の不満:「**撮影後しか枠が見えない / 撮影後ホームで予約できるか不明**」。
事実は「`/explore` は未認証でも見える（S8 実装済）」だが、**導線上でそれが伝わっていない**。本節で**見える化のための画面文言・ステータス表示**を確定する。

### 6.1 「いつ会が見えるか / いつ予約できるか」の全体像（設計の真実）
| 段階 | `/explore`（公開・会一覧） | `/browse`（ログイン後ホーム） | 応募（予約）できるか |
|------|---------------------------|-------------------------------|---------------------|
| 未ログイン | ✓ 見える | （ログイン要） | ✗（要登録） |
| ログイン済・本人確認 未提出 | ✓ 見える | ✓ 見える（応募はゲート） | ✗（本人確認が必要） |
| 本人確認 **審査中**（pending） | ✓ 見える | ✓ 見える | ✗（審査中は不可・理由表示） |
| 本人確認 **承認**（approved）+ プロフィール未登録 | ✓ 見える | ✓ 見える | ✗（プロフィール登録が必要） |
| 本人確認 **承認** + プロフィール登録済 | ✓ 見える | ✓ 見える | **✓ 応募できる** |
| 本人確認 **却下**（rejected） | ✓ 見える | ✓ 見える | ✗（再提出が必要） |

**設計の要点**: 会一覧は**どの段階でも見える**。変わるのは「応募できるか」だけ。これを各画面で**淡色のステータス + できない理由 + 次の一手**として明示する（赤エラーにしない / §8）。

### 6.2 各画面での明示（文言・ステータス）

#### (a) `/explore`（公開一覧・撮影前でも見える起点）
- 既に「登録しなくても、どんな会が開かれているかご覧いただけます。参加には登録が必要です。」がある → **維持**。
- 固定フッタの CTA は現行「登録して参加」→ **「登録して参加する」**（具体化）。押下先は `/`（ログイン）。
- **追加**: LP（`/`）からの「会を見てみる」導線（§3.2）で、撮影前にここへ来られることを体現。

#### (b) `/identity/status`（審査中・承認・却下）— 「審査中でも会は見られる」を明示
現行 pending 画面は「審査中は枠への応募はできません」+「ホームへ」（→ /browse）のみ。**会が見られることが伝わらない**。
- pending の note を**2 行**にする:
  - 「審査中は会への応募はできません」
  - **「審査中でも、開催予定の会はご覧いただけます」**（← 追加。迷子防止）
- pending のボタンを **2 つ**に:
  - Primary: **会を見てみる** → `/explore`（または `/browse`。どちらも一覧が見える）
  - Secondary: **ホームへ** → `/browse`（現行維持）
  - ※ 現行は「ホームへ」1 つ。S9 は「会を見てみる」を主に格上げ（殿の「まず見せる」方針）。
- approved 画面: 現行「続けてプロフィールを登録しましょう」+「プロフィール登録へ」→ **維持**。承認後の道筋が明確。
- rejected 画面: 現行維持（理由 + 「もう一度提出する」）。

#### (c) `/browse`（ログイン後ホーム）— 「応募できるか」をステータスで明示
- 既に未認証バナー（`◷ 本人確認が完了すると応募できます [確認する]`）がある想定（wireframes U-04）。S9 で**段階別に文言を出し分け**:
  | viewer 状態 | 上部バナー文言（淡色・責めない） | アクション |
  |-------------|----------------------------------|-----------|
  | 本人確認 未提出 | 本人確認をすると、会に応募できます | [本人確認へ進む]→ `/identity` |
  | 審査中 | 本人確認を確認中です。完了すると応募できます | [状況を見る]→ `/identity/status` |
  | 承認済・プロフ未登録 | プロフィールを登録すると、会に応募できます | [プロフィール登録へ]→ `/profile/new` |
  | 承認済・プロフ済 | （バナー無し。通常どおり応募可） | — |
  | 却下 | 本人確認をもう一度お願いします | [再提出する]→ `/identity` |
- バナーは `bg/sunken` 地 + `◷`/`✓`/`⚠` の形状（色のみ依存しない）。**赤にしない**（落ち度ではない）。

#### (d) 承認後「どこで予約できるか」の明示
殿の「撮影後ホームで予約できるか不明」への直接回答:
- 承認 + プロフ登録済になったら、**`/browse`（ホーム）の各枠カードがそのまま応募導線**になる（枠カード → `/slots/[id]` → 「この枠に応募する」）。
- これを**承認直後に一度伝える**: `/identity/status` の approved → プロフィール登録（U-02）完了後の着地で、**1 回だけ**軽い案内を出す（トースト or バナー）:
  - 文言案: **「準備ができました。ホームの会から応募できます。」** + [会をさがす]→ `/browse`。
  - これで「撮影後どこで予約？」の迷子が解消。
- 用語の統一: ユーザー向けには「**応募**」を使う（「予約」は会場予約=運営手配と紛らわしいので、ユーザーの会への参加申込は「応募」で統一）。「予約」は会場（U-08 の予約ページ）に限定。

### 6.3 本人確認の通知の行き先（明示）
- 承認 / 却下は NotificationLog（type=identity_approved / rejected）に記録、本番は LINE Messaging で本人へ。
- 各画面（U-12 提出後 / U-13）に **「結果は LINE とこの画面でお知らせします」**を `caption` で明記（現行 pending は「完了したら LINE でお知らせします」あり → 「とこの画面でも確認できます」を補う）。

## 7. 文言ガイド（ボタン・ステータスの言い換え一覧）

殿の「『次へ』等が浮く・謎文言」への横断対策。**次に何が起きるか分かる具体語**にする。

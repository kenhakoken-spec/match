## 2. ビジュアル戦略 — CSSで作る上質ビジュアル（最重要）

殿の核心要望: **「画像っぽいリッチさを、外部画像に依存せず CSS で作る」**。
方針の本質は **「線画の図解（製図）をやめ、"空気"を描く」**。庭の俯瞰を線で説明するのではなく、**やわらかな光と色のにじみ**で「あたたかく安心な庭」の気配を出す。写真が一枚も無くても成立し、将来写真が入ればその上に重ねられる構造にする。

### 2.1 ヒーロー背景 — 「あたたかい庭」のアトモスフィア（CSS具体値）

LP（`/`）と `/coming-soon` のヒーローに敷く背景。**量産SaaSの紫グラデは禁止**（design-system §8）。テラコッタ＋深緑の低彩度を、生成りの地に**淡く**にじませる。

実装イメージ（コンテナに重ねる。**色は全てトークン由来**）:

```css
/* ヒーローのアトモスフィア背景。本文の親(relative)に絶対配置 + 後ろ(z-index:-1 or 兄弟要素) */
.hero-atmosphere {
  position: absolute;
  inset: 0;
  overflow: hidden;
  pointer-events: none;
  /* 地は bg.base に溶ける */
  background-color: #FBF7F0; /* bg.base */
}

/* レイヤー1: 上部から差すあたたかい光（テラコッタ）。中心を上にずらす。 */
.hero-atmosphere::before {
  content: "";
  position: absolute;
  inset: -10% -10% auto -10%;
  height: 70%;
  background:
    radial-gradient(
      120% 90% at 30% 0%,
      rgba(194, 112, 61, 0.20) 0%,   /* accent.500 を 20% で */
      rgba(231, 183, 154, 0.12) 38%, /* accent.300 を 12% で */
      rgba(251, 247, 240, 0) 72%     /* bg.base へ透明に抜く */
    );
  filter: blur(8px);
}

/* レイヤー2: 右下に沈む落ち着いた緑の気配（庭の植栽の暗示・ごく淡く）。 */
.hero-atmosphere::after {
  content: "";
  position: absolute;
  right: -15%;
  bottom: -20%;
  width: 70%;
  height: 60%;
  background:
    radial-gradient(
      circle at 70% 70%,
      rgba(94, 122, 87, 0.14) 0%,    /* secondary.500 を 14% で */
      rgba(94, 122, 87, 0.06) 40%,
      rgba(251, 247, 240, 0) 70%
    );
  filter: blur(10px);
}
```

トーンの要点（FAIL回避）:
- **彩度は徹底して抑える**。accent は最大 0.20 透過、緑は 0.14 透過。これ以上濃くすると「ビビッド／SaaSグラデ」化する。
- **2色まで**（テラコッタ＋深緑）。3色以上の虹色グラデは禁止。青・紫は一切入れない。
- **必ず `bg.base`（生成り）へ透明に抜く**。背景が「塗り」に見えず「光のにじみ」に見えること。
- `blur(8〜10px)` でエッジを溶かす。くっきりした円は不可（"図形を置いた"感が出る）。
- スクロールで動かさない（パララックス不要・落ち着き優先）。

> なぜグラデなら良くて garden-plot SVG は不可なのか: SVG は「庭を上から見た図解」で**意味の解読**を要求し、解読できないと安っぽく見える。グラデは**解読を要求しない空気感**で、温度・安心の情緒だけを直接伝える。雑誌の扉ページの色面に近い。

### 2.2 「庭の気配」の有機的シェイプ（任意・最小限の添えのみ）

グラデだけで十分成立するが、ヒーローに**ごく控えめな有機シェイプ**を1つだけ重ねてもよい（幾何の俯瞰線画は不可）。使うなら下記の条件:
- **やわらかい曲線のブロブ（blob）**を1枚、`accent.100`（#F6E7DC）の塗り＋`opacity: 0.5`＋`blur(2px)` 程度で、ヒーロー右上 or 下に**沈ませる**。
- もしくは **円弧（arc）**1本だけ（`leaf` / `arc-light`、§3）。
- **複数置かない**。1枚まで。賑やかにすると編集感が壊れる。

blob の SVG パス例（不均一なやわらかい形・製図に見えないこと）:
```
M40,10 C70,5 95,30 90,60 C86,88 55,98 30,88 C8,79 5,45 20,28 C27,19 30,12 40,10 Z
```
viewBox `0 0 100 100`、`fill` に `accent.100`、`opacity:.5`。**stroke は付けない**（線にすると図形＝製図感が出る）。

### 2.3 カード化と奥行き（"画像っぽさ"の代替）

「画像が無く平板」への対策は**グラデ＋カードのリズム**で解決する。
- 価値ブロックを**1枚1枚のカード**（`bg.surface` + `1px line.200` + `radius.md` + `shadow.sm`）にして、平らな羅列から脱する。
- ヒーローの説明文ブロックは**地に半透明の面**（`bg.surface` を `opacity` 無しのソリッド or `rgba(255,255,255,0.7)`）を敷き、グラデの上で本文の可読性を確保しつつ「カード写真」のような佇まいにする。
- 影は **`shadow.sm` まで**（design-system §3.3）。`shadow.md` は固定要素のみ。ぷっくり浮かせない。

### 2.4 タイポgrafィの階層で「編集的・上質」に

- **主見出しは明朝（serif）でさらに大きく**: 現行 28px → ヒーロー主見出しは **32px / line-height 1.3 / letter-spacing -0.01em**（display 強化）。明朝の縦線が映える。
- **タグラインは小さく・字間を開けて**: `label` 13px / `tracking +0.08em` / `accent.600` の**小さなキャッチ**にし、主見出しを引き立てる（現行は18pxで主見出しと competing）。
- 本文は sans 15px / line-height 1.7 を維持。
- セクション見出しは現行の「細線 + 13px」を維持（編集的で良い）。
- **行長を狭める**: 本文は `max-width: 20rem`（現行どおり）。読み物の呼吸を保つ。

### 2.5 やってはいけない（このビジュアル戦略の FAIL）

- 紫・青・ピンクのグラデ → ✗（design-system §8）。テラコッタ＋深緑のみ。
- グラデの透過を 0.3 超にして「塗り」化 → ✗（光のにじみに留める）。
- くっきりした幾何シェイプ（円・三角・俯瞰の区画）を散らす → ✗（garden-plot の再来）。
- 強い影・色付き影でカードを浮かせる → ✗（`shadow.sm` まで）。
- 装飾を増やして賑やかにする → ✗（余白で見せる。迷ったら引く）。
- グラデの上に低コントラストで本文を載せる → ✗（本文は面 or 透過白の上に置きAA確保）。

## 3. BrandMotif の作り直し方針（謎SVGを何に置換するか）

### 3.1 結論

| モチーフ | 現状 | S10 の措置 | 置換先 |
|----------|------|-----------|--------|
| **`garden-plot`** | 俯瞰の製図（区画＋楕円＋↑＋○6）。**意味不明で安っぽい主因**。 | **全面撤去**。LP/ComingSoon/オンボの全使用箇所から外す。 | ヒーロー=**CSSアトモスフィア（§2.1）**。オンボ=**§7のレイアウト**（明朝見出し＋余白＋小モチーフ）。 |
| `mark` | 箱庭マーク（角丸枠＋区画＋植栽点描）。ロゴ。 | **維持**（小さく使う限り意味が通り、安っぽくない）。ただし枠の中の区画線は**省略可**（よりシンプルに）。 | そのまま（§3.3 で微調整案）。 |
| `lantern` | 石灯籠。 | **維持**（「灯り＝安心」が伝わる小アイコン）。価値ブロックで使用。 | そのまま。 |
| `gate` | 枝折戸の門。 | **維持**（「入口＝はじまり」）。 | そのまま。 |
| `stepping-stones` | 飛び石3つ。 | **維持**（「流れ」）。「ご利用の流れ」見出し脇。 | そのまま。 |
| **`leaf`（新規）** | — | **新規追加**。やわらかい葉/芽の線画1枚。「庭・成長・やわらかさ」を1ストロークで。 | ヒーローの任意の添え（§2.2）/ ComingSoon。 |
| **`arc-light`（新規・任意）** | — | 光の弧（細い円弧 + 小さな光点）。「あたたかい灯り」。`leaf` の代替で1つあれば十分。 | ヒーローの添え。 |

> 要点: **「俯瞰で説明する図（garden-plot）」をやめ、「1モチーフで情緒が伝わる小さな添え（leaf / 既存 lantern・gate）」と「CSSの空気（グラデ）」に役割分担する**。謎の解読を観者に強いない。

### 3.2 新モチーフ `leaf.svg` の仕様（frontend が起こす）

- viewBox `0 0 48 48`、`fill="none" stroke="currentColor"`、stroke 1.6、`aria-hidden`。
- **やわらかい葉が2枚、芽から伸びる**素描。製図に見えない、少し不均一な曲線。中心の葉脈は1本だけ。
- accent は最大1点（芽の先端の小さな点 or 1枚の葉脈）に `var(--brand-accent, currentColor)`。
- 線画イメージ（パス例・frontend が微調整可）:
  ```
  茎:   M24,40 C24,30 24,24 24,16
  葉左: M24,26 C16,24 11,18 12,11 C20,12 25,18 24,26 Z
  葉右: M24,22 C31,19 36,21 38,15 C31,12 26,16 24,22 Z
  芽点: M24,15 の位置に小円 r=1.4（accent）
  ```
- BrandMotif の `MotifName` union に `"leaf"` を追加、`VIEWBOX` と `SHAPES` に登録。`public/brand/leaf.svg` にも同一図形を置く（S9の二重持ち方針を踏襲）。

### 3.3 `mark`（ロゴ）の微調整（任意）

現行 mark は問題ないが、より洗練させるなら:
- 枠内の十字の区画線（`line` 2本）を**外すか1本に減らす**と、より記号的でロゴらしくなる。
- 植栽点描・下部の3点は残す（箱庭＝小さな庭の含意）。
- これは任意。**現行維持でも可**（garden-plot ほどの問題はない）。

### 3.4 撤去の具体（garden-plot をどこから外すか）

| 箇所 | 現状 | S10 |
|------|------|------|
| `LoginScreen.tsx` ヒーロー | `<BrandMotif name="garden-plot" ... max-w-[20rem]>` | **削除**。§4 のヒーロー（CSSアトモスフィア＋見出し）に置換。 |
| `ComingSoon.tsx` ヒーロー | 同上 | **削除**。§6 のヒーローに置換。 |
| `onboarding/page.tsx` 説明スライド1 | `SLIDES[0].motif = "garden-plot"` | **`"leaf"` に変更**（§7）。 |
| `onboarding/page.tsx` 性別ステップ | `<BrandMotif name="garden-plot" ...>` の `aspect-[4/3]` 箱 | **削除**し編集的レイアウトに（§7.2）。 |
| `public/brand/garden-plot.svg` | ファイル存在 | **削除**（参照ゼロにしてから）。 |
| `BrandMotif.tsx` `SHAPES["garden-plot"]` / `VIEWBOX` / union | 定義あり | **削除**（`MotifName` から `"garden-plot"` を外す）。 |

## 4. LP（`/` LoginScreen）全面リデザイン

ペルソナ: 20代後半〜30代後半・東京（恵比寿/池袋/銀座）・1対1アプリ疲れ・安全/本人確認重視・気まずさ回避。
マーケ目標: **ファーストビューで「誰のための・何が違う」が3秒で伝わる**。不安解消（本人確認必須／3対3で安心／真剣な人だけ）を前面に。具体性（エリア・曜日）で実在感。

### 4.1 情報設計（縦の重みづけ・スクロール順）

スマホ縦・単一カラム。**固定フッタは廃止**（§5）。上から:

1. **ブランドロックアップ**（マーク＋HAKO-NIWA＋箱庭・エリア）— 軽く
2. **ヒーロー（ファーストビュー）**: CSSアトモスフィア背景の上に
   - タグライン（小さなキャッチ）
   - **主見出し（明朝・大）**
   - **サブ見出し1文（誰のための/何が違う）** ← ★新規。3秒理解の核
   - **主CTA（LINEではじめる）＋副CTA（会を見てみる）** ← ★ヒーロー内に置く（被り解消の主眼）
   - LINE外案内の小リンク
3. **不安解消の価値ブロック**（4点・カード化）← ベネフィット言語。ペルソナの不安に1:1で応答
4. **ご利用の流れ**（5ステップ）
5. **具体（エリア・曜日）ブロック** ← ★新規。実在感（恵比寿/池袋/銀座・水金土 19:30）
6. **末尾CTA再掲**（主＋副）← スクロールしきった人を取りこぼさない
7. **規約・特商法リンク**（小さく）

> 量産LP回避（design-system §8）: 巨大ヒーロー＋3カラム特徴＋キラキラにしない。価値は**縦に積むカード**、各カードに線アイコン＋見出し＋1〜2行。

## 5. 固定CTA被りの解消（レイアウト構造）

### 5.1 問題の構造（再掲）

現行 LoginScreen は `fixed inset-x-0 bottom-0` のフッタに error + 主CTA + 副CTA + 規約2段を詰め、本文側を `pb-44`(176px) で逃がしている。だがこのフッタは実機で**約200px超**になり、`pb-44` では足りず本文（「男女3人ずつ…」）に被る。動的フォント拡大・小型端末でさらに悪化する。

### 5.2 解決方針: **固定フッタを廃し、CTAをページフローへ**

- **主CTAをヒーロー内に置く**（§4.2 / §4.5）。ファーストビューで即押せる＝マーケ的にも最善。
- **末尾にもCTAを再掲**（§4.2）。長いLPでスクロールしきった人の取りこぼし防止。
- これで `position: fixed` が不要になり、**被りは構造的に発生しない**。`pb-44` も廃止（`main` の `pb` は通常の `pb-12` 程度でよい）。
- error 表示は**主CTAの直上**（ヒーロー内）に置く（押した場所の近くで結果が見える）。

### 5.3 任意: スクロール追従の細バー（採用は任意・本文を隠さないこと）

長いLPで「常に主CTAを出したい」需要があれば、**本文を隠さない**条件下でのみ許可:
- バーは **主CTA 1個のみ**（副CTA・規約はバーに入れない）。高さは**ボタン高+上下8px ＝ 約68px**まで。
- 背景は**半透明**（`rgba(251,247,240,0.88)` ＝ bg.base 88% + `backdrop-blur-[6px]`）＋上 `1px line.200` ＋ `shadow.md`。
- **ヒーローのCTAが画面内にある間は出さない**（IntersectionObserver でヒーローCTAが view 外に出たら fade-in）。これで「最初からCTAが本文に被る」事故が起きない。
- 本文側の `pb` はバー高さぶん（`pb-24` ≒ 96px）確保。
- **この細バーは必須ではない**。§5.2（フロー埋め込み＋末尾再掲）だけで殿の被り問題は解決する。実装簡素さを優先するなら細バーは省略してよい。

> 推奨: **まず §5.2 のみで実装**（固定廃止＋ヒーロー＋末尾再掲）。実機確認で「CTAが遠い」と感じたら §5.3 の追従バーを追加。最初から固定パネルに戻さないこと（被りの再発防止）。

## 8. ログインUX（エラー文言・再試行・LINE外案内）

殿報告「LINEではじめるでエラー」「ログイン外で開いた人が詰まる」への対策。**ログイン遷移ロジック自体（runLogin / 自動再開 useEffect）は挙動不変で維持**。変えるのは**文言・案内・配置**のみ。

### 8.1 LINE外案内の常設（詰まり防止の主眼）

- ヒーローの主CTA直下に、**エラーが無くても常時**小さく案内を出す（§4.2）:
  - 文言: **スマホのLINEで開くと、そのまま進めます**（caption / ink-500）。
- これにより「PCブラウザ／LIFF外で開いた人」が、押す前に前提を理解できる。
- 既存の自動再開（LINE同意から戻った時の `runLogin`）は維持。

### 8.2 エラー時の文言（原因別・必ず「次の一手」を添える）

現行 `errorMessage()` は原因別に分かれているが、**「次にどうすればいいか」が弱い**。各文言に一手を足す:

| エラーコード | 現行文言 | S10 文言（原因＋次の一手） |
|-------------|---------|---------------------------|
| `no_id_token` | LINEのIDトークンを取得できませんでした。LINEアプリ内から開いているかご確認ください。 | **LINEの確認情報を取得できませんでした。お手数ですが、スマホのLINEアプリ内からこのページを開いて、もう一度お試しください。** |
| `liff_init_failed*` | LINE連携の初期化に失敗しました。LINEアプリ内から開いてお試しください。 | **LINE連携の準備に失敗しました。通信環境をご確認のうえ、スマホのLINEアプリ内から開き直してお試しください。** |
| `token_exchange_failed*` | LINEの確認に失敗しました（トークン検証エラー）。時間をおいて再度お試しください。 | **LINEの確認に失敗しました。少し時間をおいて、もう一度「LINEではじめる」を押してください。** |
| 既定 / その他 | ログインに失敗しました。もう一度お試しください。 | **ログインに失敗しました。もう一度「LINEではじめる」を押してお試しください。** |

- トーンは design-system §0（誠実・責めない・煽らない・感嘆符なし）。
- エラーブロックの見た目は **`state/warn`系**（現行どおり `state-warn` の淡い橙地）。**赤（danger）にしない**（ユーザーの落ち度ではなく環境要因が主なため。design-system §5 と整合）。
- 配置: **主CTA の直上**（ヒーロー内）。固定フッタ廃止に伴い、押下箇所の近くで結果が見える。

### 8.3 再試行の導線

- エラー後は **主CTA「LINEではじめる」がそのまま再試行ボタン**になる（loading 解除済み）。文言で「もう一度…押してください」と誘導（§8.2）。
- 追加ボタンは設けない（1画面1プライマリの原則・design-system §4.1）。
- loading 中の文言は現行「接続しています…」を維持。

### 8.4 LINE外で開いた可能性が高い時の補助（任意）

- `no_id_token` / `liff_init_failed` は **LINE外で開いた可能性が高い**エラー。これらの時のみ、案内文をやや強調（caption → body / ink-700）してもよい（任意）。
- QRコード等の追加は**スコープ外**（本書では文言案内に留める。殿要件は「案内の出し方」であり、QR実装は別途判断）。

## 9. 新トークン追記（CSSビジュアル用ユーティリティ）

§2 のアトモスフィアは**既存カラートークンの透過**で作るため、**新しい色トークンは追加不要**。ただし frontend が実装しやすいよう、以下を `globals.css` のユーティリティ（または Tailwind の `@layer components`）として定義することを推奨:

```css
@layer components {
  /* ヒーローのあたたかい庭のアトモスフィア背景(§2.1)。装飾専用・aria-hidden 前提。 */
  .hero-atmosphere {
    position: absolute;
    inset: 0;
    overflow: hidden;
    pointer-events: none;
    background-color: #FBF7F0; /* bg.base */
  }
  .hero-atmosphere::before {
    content: "";
    position: absolute;
    inset: -10% -10% auto -10%;
    height: 70%;
    background: radial-gradient(
      120% 90% at 30% 0%,
      rgba(194,112,61,0.20) 0%,
      rgba(231,183,154,0.12) 38%,
      rgba(251,247,240,0) 72%
    );
    filter: blur(8px);
  }
  .hero-atmosphere::after {
    content: "";
    position: absolute;
    right: -15%;
    bottom: -20%;
    width: 70%;
    height: 60%;
    background: radial-gradient(
      circle at 70% 70%,
      rgba(94,122,87,0.14) 0%,
      rgba(94,122,87,0.06) 40%,
      rgba(251,247,240,0) 70%
    );
    filter: blur(10px);
  }
  /* オンボ用の弱いアトモスフィア(上1/3・緑なし・任意)。 */
  .hero-atmosphere--soft::after { display: none; }
  .hero-atmosphere--soft::before {
    height: 40%;
    background: radial-gradient(
      110% 80% at 50% 0%,
      rgba(194,112,61,0.12) 0%,
      rgba(251,247,240,0) 70%
    );
  }
}
```

- 透過値の RGB は全てトークン由来（accent.500 `#C2703D`=194,112,61 / accent.300 `#E7B79A`=231,183,154 / secondary.500 `#5E7A57`=94,122,87 / bg.base `#FBF7F0`=251,247,240）。
- Tailwind 任意値（`bg-[radial-gradient(...)]`）で直接書いてもよいが、可読性のため `globals.css` のユーティリティ化を推奨。
- **新トークン追加は不要**（design-system §7 の既存トークンで完結）。`tailwind.config.ts` は変更不要。

## 2. PC最適化（レスポンシブ）方針 — app-shell の解放

### 2.1 根本原因と設計判断

現状 `layout.tsx` は全 children を `.app-shell`（`max-width:480px; margin-inline:auto`）で包む。これが**全画面を 480px に固定**している主因。admin だけは配下で `w-full lg:grid` を組むが、親が 480px なので**本来は admin も詰まる**（admin が広く見えているのは、admin 配下が `min-h-[100dvh] w-full` で親いっぱい＝480px いっぱいに広がっているだけで、実は 480px 内。Shogun の「admin は PC前提でOK」は機能面の話。本書では admin は対象外＝触らないが、**app-shell を解放しても admin は壊れない**よう後述の方針を取る）。

**設計判断**: `.app-shell` の 480px 固定を**廃し**、代わりに「**広げてよい画面だけが内側で広がる**」構造にする。これが「モバイルは壊さない・PCは映える」を両立する最も安全な形。

- 理由: `.app-shell` を単純に広げると、**フォーム/オンボ/詳細/mypage まで横に間延び**して可読性が落ちる（編集的余白の崩壊・design-system §3.4）。逆に 480px のままだと PC で全画面が詰まる。→ **解放（外側）＋各画面で適切な内側コンテナ**の二段構えが正解。

### 2.2 app-shell の変更（globals.css）

`.app-shell` を**画面いっぱい（max-width 撤廃）**にし、横方向の中央寄せは各画面 or 内側コンテナに委ねる。ただし「何も指定しない画面が PC でベタ全幅になる」事故を防ぐため、**既定の内側最大幅を持つユーティリティ**を用意し、各画面はそれを使うか、独自に広げる。

```css
/* 変更後: app-shell は幅制約を持たない器に降格。中央寄せ・最大幅は内側で決める。 */
.app-shell {
  /* max-width: 480px; を撤廃 */
  margin-inline: auto;        /* 子が max-width を持つ時に中央寄せ */
  min-height: 100dvh;
  width: 100%;
}

@layer components {
  /* モバイル幅を保つ画面用(オンボ/フォーム/詳細/mypage/applications 等)。
     LIFF 主体の縦長UIは PC でも 480px の読み物幅を維持する(間延び防止)。 */
  .shell-narrow {
    max-width: 480px;
    margin-inline: auto;
    width: 100%;
  }
  /* 広げてよい画面の本文コンテナ(LP/explore/browse)。md で広がり lg で頭打ち。 */
  .shell-wide {
    max-width: 480px;          /* base: モバイルは現状維持 */
    margin-inline: auto;
    width: 100%;
  }
}
```

> 注: Tailwind 任意値だけでも実装可能（`mx-auto w-full max-w-[480px] md:max-w-3xl lg:max-w-5xl`）。可読性のため `.shell-narrow` / `.shell-wide` をユーティリティ化しておくと、各 `main` に1クラス付けるだけで意図が明確になる。**どちらで実装してもよい**（frontend 裁量）。`shell-wide` の md/lg 拡張は §3・§4 で各画面ごとに指定する（ユーティリティに breakpoint を埋めず、各画面の `main` で `md:max-w-*` を付ける方が画面ごとに最適化しやすい。**推奨は後者**）。

### 2.3 移行の安全策（モバイルを壊さないための鉄則）

- **base（〜767px）の見た目は一切変えない**。すべての widen 指定は `md:` 以上の prefix を付ける。`md` 未満で発火するスタイルを足さない。
- app-shell の `max-width:480px` を外す影響は、**各画面が自分の最大幅を持っていれば base では同一**（480px に収まる）。よって全画面の `main`（or 直下ラッパ）に **base で `max-w-[480px] mx-auto`** が乗るようにするのが最優先（これが抜けると PC でベタ全幅になる画面が出る）。
- 影響範囲は全ルートの最上位ラッパ。**各画面に1クラス（`shell-narrow` か widen 指定付き `shell-wide`）を付ける**機械的作業。漏れがあると「その画面だけ PC で全幅」になるので、qa で全ルート目視（§9）。

### 2.4 画面ごとの幅ポリシー（一覧）

| 画面 | base(〜767) | md(768〜1023) | lg(1024〜) | 内側構成 |
|------|-------------|---------------|-----------|----------|
| **LP（`/` LoginScreen）** | 480px 1カラム | **広げる**（コンテンツ最大 約 720〜768px、Hero 2カラム化） | 最大 約 960px・Hero 2カラム維持 | §3 |
| **explore（公開一覧）** | 480px 1カラム | **2列グリッド**（最大 約 720px） | **3列**（最大 約 960px） | §4 |
| **browse（認証一覧）** | 480px 1カラム | **2列グリッド** | **3列** | §4 |
| 詳細（slots/[id]・explore/[id]） | 480px | **480px 維持**（読み物・申込フローは1カラムが最適） | 480px 維持（任意で 560px まで可） | §2.5 |
| オンボ（onboarding） | 480px | 480px 維持 | 480px 維持 | 触らない |
| mypage / applications / profile / 各フォーム | 480px | 480px 維持（or 560px まで） | 同左 | `shell-narrow` |
| identity / 決済 / 評価 等のフロー画面 | 480px | 480px 維持 | 同左 | `shell-narrow` |
| coming-soon | 480px | 480px 維持（任意で LP と同じ widen でも可） | 同左 | `shell-narrow`（任意 widen） |
| admin | （対象外） | （対象外・現行のまま） | （対象外） | 触らない |

> 原則: **「一覧性・回遊性が価値の画面（LP/explore/browse）だけ広げる」**。意思決定や入力に集中させたい画面（詳細・フォーム・オンボ）は **480〜560px の読み物幅を守る**（広げると視線移動が増え、フォームが間延びして使いにくくなる＝UX劣化）。これは「PC で映える」と矛盾しない（広い画面でも中央の最適幅カラムは上質に見える＝雑誌の本文段組）。

### 2.5 詳細画面を広げない理由（明示）

slots/[id] / explore/[id] は「日付主役ヘッダ → 募集状況 → 参加条件 → 料金 → 当日の流れ → 応募フッタ」の**縦の意思決定フロー**。これを2カラム化すると視線が分断され、応募という主目的が弱まる。**480px（任意で lg のみ 560px）を維持**し、PC では「中央に上質な1枚の案内が置かれる」見え方にする。背景の余白が気になるなら §7.4 の「PC背景処理」で寂しさを消す。

## 3. LP のレスポンシブレイアウト（モバイル / md / lg）

LP は最も「映え」が問われる画面。モバイルは現行（s10/s11）を維持し、**md 以上で「左=コピー＋CTA / 右=大きい Hero」の2カラムヒーロー**に再構成する。これは DINE 等の洗練した縦型サービスが PC で見せる定番で、**安っぽくならず情報の主従も保てる**。

### 3.1 ブレークポイント別の全体構成

**base（〜767px・現行維持）**: 1カラム縦積み。
```
ロックアップ → ヒーロー[ Hero絵 → タグライン → 主見出し → 説明カード → CTA → LINE外案内 ]
→ 価値4カード → 流れ → 開催について → 末尾CTA → 規約
```

**md（768〜1023px）**: ヒーローを**2カラム**化。それ以降のセクションは**幅を広げ、価値カードを2列**に。
```
┌───────────────────── shell-wide (max ~720px) ─────────────────────┐
│  [mark] HAKO-NIWA ・ 箱庭 ・ 東京 恵比寿/池袋/銀座                  │  ← ロックアップ(中央 or 左)
│ ╭─────────────── ヒーロー(2カラム) relative isolate ───────────╮ │
│ │ ┌── 左:コピー+CTA(約46%) ──┐  ┌── 右:Hero絵(約54%) ──────┐ │ │
│ │ │ みんなが出会える場所      │  │                          │ │ │
│ │ │ 3対3で、会いにいく。(明朝)│  │   ▓▓ HeroScene ▓▓        │ │ │  ← 右に大きく
│ │ │ ┌────────────────────┐  │  │  (空/光/庭/灯/6人影)     │ │ │
│ │ │ │ 説明文(白面カード)   │  │  │  高さ 左カラムに追従      │ │ │
│ │ │ └────────────────────┘  │  │                          │ │ │
│ │ │ [ LINEではじめる ]       │  │                          │ │ │
│ │ │ ( 会を見てみる )         │  └──────────────────────────┘ │ │
│ │ │ スマホのLINEで…(caption) │                                │ │
│ │ └──────────────────────────┘                                │ │
│ ╰──────────────────────────────────────────────────────────────╯ │
│ ── 不安がいらない理由 ──                                           │
│ ┌────────────┐ ┌────────────┐                                    │  ← 価値カード 2列
│ │ 価値①      │ │ 価値②      │                                    │
│ └────────────┘ └────────────┘                                    │
│ ┌────────────┐ ┌────────────┐                                    │
│ │ 価値③      │ │ 価値④      │                                    │
│ └────────────┘ └────────────┘                                    │
│ ── ご利用の流れ ──   ── 開催について ──                            │  ← 任意で2カラム(流れ|開催)
│ 末尾CTA(中央・最大 ~520px) / 規約                                  │
└────────────────────────────────────────────────────────────────┘
```

**lg（1024px〜）**: コンテンツ最大 **約 960px**。ヒーロー2カラムは維持（左右比はそのまま）、価値カードは **2列のまま**（4枚なので3列にすると1枚あふれて不格好。2×2 が最も整う）。Hero 絵は右カラムでさらに大きく見える。

### 3.2 ヒーロー2カラムの実装指定（md+）

現行 `LoginScreen.tsx` のヒーロー（§既存 96〜147行）を、**md 以上で2カラム**にする。base は現行のまま（Hero 絵が上、コピーが下）。

- ヒーロー section: 現行 `relative isolate mt-10 overflow-hidden` は維持。内側に**グリッド**を追加:
  - base: 現行どおり Hero 絵 → テキストの縦並び。
  - md+: `md:grid md:grid-cols-[minmax(0,46fr)_minmax(0,54fr)] md:items-center md:gap-8`。
  - **左カラム**（コピー＋CTA）: タグライン → 主見出し → 説明白面カード → エラー → CTA → LINE外案内（現行の並び）。
  - **右カラム**（Hero 絵）: `HeroScene`。base では左カラムの**上**に出るので、DOM順は「Hero絵 → コピー」のままにして、md+ で `md:order-2`（Hero）/ `md:order-1`（コピー）で左右を入れ替える。**または** DOM順を「コピー → Hero」にして base で `order` を入れ替える。**推奨**: DOM は現行（Hero→コピー）維持、md+ で order 指定（base の見た目を1px も変えないため）。
- **主見出しの拡大**（md+）: base `text-[32px]` → `md:text-[40px] lg:text-[44px]`、`leading-[1.15]`。PC で明朝の主見出しが効く。
- 説明白面カード: base `max-w-[20rem]` → md+ は左カラム幅に追従させるため `md:max-w-none`。
- CTA（md+）: 2ボタン縦積みのままでよいが、**横幅が広がりすぎると押しにくい/間延びする**ため、左カラム内で `md:max-w-[20rem]`（ボタン群だけ幅を抑える）。主CTA・副CTA とも幅はこのコンテナに従う。
- Hero 絵の大きさ（md+ 右カラム）: `w-full`。viewBox 比（後述 §6 で 360×300 に変更）で、右カラム幅 約 380〜500px なら高さ 約 320〜420px の堂々とした絵になる。**枠線/影なし**（現行どおり地に溶かす）。

### 3.3 ロックアップの扱い（md+）

- base: 現行どおり左上。
- md+: ヒーローの**上に幅いっぱい**で置いたままでよい（中央寄せ or 左寄せ）。2カラムの上に1行で乗る形。**ロックアップを左カラムに押し込めない**（窮屈になる）。現行の `BrandLockup` をヒーロー grid の**外・上**に残す。

### 3.4 価値カード・流れ・開催・末尾CTA（md+）

- **価値カード（ValueList）**: base 1列（`space-y-3`）→ md+ 2列。`ul` を `md:grid md:grid-cols-2 md:gap-3 md:space-y-0` に。4枚が 2×2 で整う。**lg でも2列維持**（3列にしない。4枚は2×2が最適）。
- **流れ（FlowList）+ 開催（ConcreteBlock）**: md+ で**横2カラム**にしてもよい（`md:grid md:grid-cols-2 md:gap-8`）。任意。縦に積んだままでも可（実装簡素優先なら縦のまま・幅だけ広がる）。**推奨: md+ で 流れ｜開催 の2カラム**（縦の冗長感が減り、PC で締まる）。
- **末尾CTA**: 幅いっぱいだとボタンが間延びするので、`md:max-w-[520px] md:mx-auto` で中央に抑える。見出し「まずは、のぞいてみませんか。」は中央維持。
- **規約**: 現行どおり中央・小さく。

### 3.5 LP の main ラッパ（幅指定の実体）

現行 `<main className="flex min-h-[100dvh] flex-col px-6 pb-12 pt-16">`。これに widen を足す:

```
<main className="mx-auto flex min-h-[100dvh] w-full max-w-[480px] flex-col px-6 pb-12 pt-16
                 md:max-w-3xl md:px-8 lg:max-w-5xl">
```
- `max-w-[480px]`（base）→ `md:max-w-3xl`(768px) → `lg:max-w-5xl`(1024px)。
- `px-6`(24px) → `md:px-8`(32px)。PC で左右に少し余白を足し、本文が画面端に貼り付かないように。
- これで app-shell の 480px 撤廃後も base は 480px のまま、PC で段階的に広がる。

## 4. explore / browse のレスポンシブ（md以上で2〜3列）

一覧は「どんな会があるか一望する」のが価値。PC では**カードを2〜3列グリッド**にして一覧性を上げる。モバイルは現行1列を維持。

### 4.1 リスト表示のグリッド化

現行はどちらも `<ul className="space-y-3">`（1列縦積み）。これを**グリッド**に:

```
<ul className="grid grid-cols-1 gap-3 md:grid-cols-2 lg:grid-cols-3" data-testid="...">
  {sorted.map(...)}
</ul>
```
- base: `grid-cols-1`（＝現行と同じ1列・`gap-3` は `space-y-3` と等価の間隔）。
- md: `md:grid-cols-2`。
- lg: `lg:grid-cols-3`。
- カード（SlotCard / PublicSlotCard）は**現行のまま**でグリッドのセルに収まる（カードは `block` で幅100%なのでセル幅に追従）。**カード内部のレイアウト変更は不要**。日付主役ブロック・充足・条件/料金はそのまま。
- **カードの高さ揃え**: グリッドの行内でカード高さがばらつくと不格好。`ul` に `md:auto-rows-fr`（or 各 `li` を `h-full`、カードに `h-full`）を付け、**同一行のカードを同じ高さに**伸ばす。SlotCard/PublicSlotCard のルート `Link` に `h-full` を足し、内部は現行のまま（上が日付、下が料金で、余白は中段に入る）。**任意だが推奨**（揃うと一気に「ちゃんとした」印象になる）。

### 4.2 explore のページ幅と固定CTAの扱い

explore（`(public)/explore/page.tsx`）:
- `main` に widen: 現行 `<main className="flex-1 px-5 pb-28 pt-4">` → `mx-auto w-full max-w-[480px] md:max-w-3xl lg:max-w-5xl md:px-8` を付与（`flex-1 px-5 pb-28 pt-4` は維持）。
- 親 `<div className="flex min-h-[100dvh] flex-col">` は維持。
- **固定 RegisterCta フッタ**（`sticky bottom-0 ... border-t bg-bg-surface px-5 py-3 shadow-md`）: PC で画面幅いっぱいの帯になると間延びする。**中身を中央コンテナで抑える**: フッタ直下の内容を `mx-auto w-full max-w-[480px] md:max-w-3xl lg:max-w-5xl md:px-8` でラップ（帯自体は全幅、CTA ボタンは本文と同じ最大幅に揃う）。これで本文カラムとCTAの左右が揃い、整然とする。
- 説明文 `p`（「登録しなくても…」）と ViewToggle・グリッドはこの widen された main 内に入るので自動で広がる。ViewToggle は現行 `max-w-[16rem]` 中央のまま（広げない・トグルは小さくてよい）。

browse（`(tabs)/browse/page.tsx`）:

- 親はタブ layout（`(tabs)/layout.tsx`）。BottomTabs は `sticky bottom-0` で全幅の帯。PC でタブが間延びするので、§4.3 で対応。

### 4.3 タブ layout（BottomTabs）の PC 最適化

`(tabs)/layout.tsx` と BottomTabs は本体の主要回遊。PC で BottomTabs が画面幅いっぱりに広がると、3タブが両端に離れて間延びする。

- **BottomTabs の中身を中央コンテナで抑える**: `nav`（`sticky bottom-0 z-10 grid grid-cols-3 ...`）の**内側**を `mx-auto w-full max-w-[480px]` でラップ。帯（border-t・bg-surface）は全幅、3タブのグリッドは 480px に収まり中央に。これで PC でもタブが詰まらず自然。
- `(tabs)/layout.tsx` の `<div className="flex flex-1 flex-col">` は children をそのまま流す（各ページ側の main が widen を持つ）。layout 側に幅制約は足さない（ページごとに幅ポリシーが違うため）。
- **base は現行のまま**（タブは元々 480px 内なので変化なし）。

> 注意: BottomTabs を中央 480px に抑えると、**広い一覧（lg 3列・960px）の下でタブだけ 480px 中央**になる。これは不自然ではない（タブはモバイル的なナビなので中央にまとまっていて良い）が、もし「一覧幅にタブも合わせたい」なら BottomTabs ラッパも `md:max-w-3xl lg:max-w-5xl` にする。**推奨: タブは 480px 中央のまま**（ナビは指の届く中央が自然・回遊一覧とは役割が違う）。frontend 裁量。

## 5. カレンダーの PC 最適化

`SlotCalendar.tsx` はスマホ縦前提（7列・各セル `h-11 w-11`）。PC では横に間延びしないよう、**カレンダー本体に最大幅**を与え、**選択日の会カードはグリッド化**する。

### 5.1 カレンダー本体（月グリッド）

- カレンダー全体（`SlotCalendar` のルート `<div>`）に **`mx-auto w-full max-w-[420px]`** を base から付ける（カレンダーは元々これ以上広げても1セルが巨大になるだけで見にくい）。
- md+ でやや広げてもよい: `md:max-w-[520px]`。セルは `h-11 w-11`（44px）→ md+ で `md:h-12 md:w-12`（48px）にすると PC で押しやすく見やすい。**任意**（base のままでも可）。
- 月送りヘッダ・曜日見出し・日セルの**構造とトークンは現行維持**（曜日色・accentドット・選択日塗り・今日 ring）。

### 5.2 選択日の会カード（カレンダー直下）

現行 `<ul className="mt-3 space-y-3">`（1列）。カレンダー本体が中央 ~520px なのに対し、**選択日の会は md+ で2列**にすると一覧性が上がる:

- 選択日見出し `h3`（`6/19(木) の会`）は現行のまま。
- その下の `ul` を **`md:grid md:grid-cols-2 md:gap-3 md:space-y-0`** に（base は `space-y-3` 維持）。
- ただしカレンダー本体（~520px 中央）と会カード（2列で ~720px）の**幅がずれる**と不揃いに見える。対策: **選択日の会のコンテナを、カレンダー本体とは別に main 幅（md:max-w-3xl）いっぱいまで広げてよい**。つまりカレンダーの月グリッドだけ中央 520px、会カードは下で2列に広がる、という構成。
  - 実装: `SlotCalendar` 内で「月グリッド部」を `mx-auto max-w-[520px]`、「選択日の会部」は `max-w` を付けず親（main）の幅に従わせる。
- **lg でも会カードは2列**でよい（カレンダー文脈なので3列にすると賑やかすぎ。2列が落ち着く）。リスト表示（§4.1）は3列、カレンダー直下は2列、と差をつけるのは意図的（カレンダーは「その日の数件」を見る文脈で件数が少ない）。

### 5.3 会ゼロ月・空状態（PC）

## 6. Hero ビジュアルの「映え」強化（最重要・推奨手段と具体仕様）

殿の最重要要望: **「もっと画像を入れて映えさせる」**。現状 `HeroScene` は淡くコントラストが弱く主役感が不足。**作り込みを一段引き上げる**。

### 6.1 手段の比較と推奨（再確認）

| 手段 | 内容 | 判定 |
|------|------|------|
| **(a) HeroScene の SVG 作り込み強化** | 空グラデの階調増・光源強化・人影コントラストUP・前景レイヤー追加（敷石/低木）で奥行き。viewBox を縦に伸ばして堂々と。 | **★主軸（採用）** |
| **(b) 控えめなCSS演出** | 額装（ごく薄い内側シャドウ/ヴィネット）・微グレイン（任意）。SVG を引き立てる範囲で。 | **★補助（採用・最小限）** |
| (c) raster画像（写真/AI生成） | 実写 or 生成画像を配置。 | **不採用（当面）**。理由↓ |

**(c) raster を当面採用しない理由**（殿のライセンス懸念への直答）:
1. **ライセンス継続管理リスク**（フリー素材も規約変更・帰属表示・商用条件の管理が必要。マッチングという信頼性が肝の領域で素材トラブルは致命的）。
2. **肖像・年齢・民族のズレ**（無料人物写真は世界観＝20後半〜30後半・東京・等身大に合うものが実務上見つけにくい。借り物感が出る）。
3. **AI生成画像は「AIっぽさ排除」の方針（design-system §0）と正面衝突**。手指・顔の破綻リスク、生成物の権利の不透明さもある。
4. **配信/CLS リスク**（外部 `<img>` は読込失敗で枠だけ＝過去の事故。インラインSVGは外部依存ゼロで必ず描画）。
5. **将来の逃げ道は確保**: HeroScene を `<img>` に差し替え可能な構造（§6.5）にしておけば、**自前撮影 or 許諾済み写真が用意できた時点で raster へ移行**できる。今は SVG を作り込んで「映え」を出す。

> 結論: **「映え」は raster でなくとも、SVG の階調・光・奥行き・人影の存在感 ＋ 控えめなCSS額装で十分に出せる**。むしろトークン統制下で「安っぽくない・ちゃんとした・おしゃれ」を確実に作れる。

### 6.2 HeroScene 強化の具体仕様（frontend が作る）

**viewBox を `0 0 360 280` → `0 0 360 300`** に変更（縦に20伸ばし、空と前景の余白を確保＝詰まり感の解消）。`preserveAspectRatio` は **`xMidYMid meet`** に変更（現行 `slice` は端を切るため、`meet` で絵全体を見せる。LP では絵全体が見えた方が「1枚絵」として映える）。

> base（モバイル1カラム）では Hero 絵の高さが現行より少し増える（meet で全体表示＋viewBox縦増）。これは「映え」要望に沿う（主役を大きく）。詰まり過ぎないよう、LP base のHero枠は高さ自動（`h-auto w-full`）のまま。

#### レイヤー構成（後→前・現行4層を6層に拡張）

**色は全て design-system §7 のトークン由来。紫・青・原色なし。** hex は §1 トークンから。

**① 空（縦グラデ・階調を増やす）** — 現行2stop → **4stop** で夕暮れの深みを出す:
```
linearGradient id="hs-sky" x1=0 y1=0 x2=0 y2=1
  stop 0    #F0D9C4   (accent-100 #F6E7DC を僅かに濃く＝上空の焼け。※後述の新トークン不要案も)
  stop 0.42 #F6E7DC   (accent-100)
  stop 0.72 #FBF7F0   (bg-base)
  stop 1    #FBF7F0   (bg-base)
```
- **新色を足さないなら**: stop0 を `#F6E7DC`(accent-100) のままにし、代わりに**光の弧（②）を強める**ことで上空の焼けを表現する。`#F0D9C4` は accent-100 と accent-300 の中間色で「新トークンではなく Hero 内のグラデ中間値」として許容範囲（design-system のパレットを増やすわけではなく、絵の中の階調）。**推奨: 新トークンは追加せず、stop0=`#F6E7DC`／光の弧②で深みを出す**（§8 の方針＝新トークン最小）。ただし「もっと映え」を強く出すなら Hero 内ローカルに `#F0D9C4` を1箇所だけ使うのは可（パレット追加ではない）。

**② 沈む陽の光（強化・主光源）** — 現行 `accent-300` 透過 → **不透明度と芯を上げ、主光源として明確に**:
```
radialGradient id="hs-glow" cx=0.5 cy=0.42 r=0.6
  stop 0    #E7B79A stopOpacity 0.95   (accent-300・中心は強く)
  stop 0.45 #E7B79A stopOpacity 0.45
  stop 1    #E7B79A stopOpacity 0
配置: ellipse cx=210 cy=104 rx=170 ry=140 fill=url(#hs-glow)
陽の芯(二重円・現行より明確に):
  circle cx=210 cy=98 r=26 fill #E7B79A opacity 0.65
  circle cx=210 cy=96 r=13 fill #C2703D opacity 0.55   (accent-500・芯)
  circle cx=210 cy=95 r=6  fill #C2703D opacity 0.75   (芯のコア・小さく濃く)
```
- これで「夕日がちゃんとある」＝映えの核。**ハードエッジ厳禁**（radialGradient で縁を溶かす・現行方針維持）。

**③ 遠景の稜線（新規・奥行きの最奥）** — 空と庭の間に**遠い丘/木立のシルエット**を1枚足して奥行きを深く:
```
path(なだらかな低い稜線) d="M0 188 C 90 178, 180 184, 270 176 C 320 172, 345 178, 360 174 L360 210 L0 210 Z"
  fill #5E7A57 opacity 0.12   (secondary-500・ごく淡く＝遠さ)
```
- 遠景は淡く・低彩度（空気遠近法）。これが入ると「平板」が解消する。

**④ 庭の丘（強化・現行2枚を維持しつつコントラスト微増）**:
```
奥の丘: 現行 path fill #E7EDE6 opacity 0.9 → opacity 1.0 (secondary-100・しっかり)
手前の丘: linearGradient hs-hill-front
  stop 0 #5E7A57 stopOpacity 0.38   (現行0.30→0.38・手前を少し締める)
  stop 1 #5E7A57 stopOpacity 0.18   (現行0.16→0.18)
  path 形状は現行維持(不均一ベジェ)
```

**⑤ 前景の添え（新規・最前面の地面ディテール）** — 奥行きの「手前」を作り、人影の足元を支える:
```
敷石(stepping-stones を絵に溶かす): 手前中央に楕円2〜3枚を低彩度で
  ellipse cx=150 cy=268 rx=26 ry=7  fill #5E7A57 opacity 0.16
  ellipse cx=205 cy=276 rx=22 ry=6  fill #5E7A57 opacity 0.14
  ellipse cx=120 cy=280 rx=18 ry=5  fill #5E7A57 opacity 0.12
低木(左右の隅にやわらかい塊・人影の引き立て):
  左下: path(やわらかい塊) fill #5E7A57 opacity 0.20 ＋ 上に #E7EDE6 のハイライト塊 opacity 0.5
  右下: 同様に小さく
```
- 前景は人影より**手前か同列**だが、**人影の存在感を消さない**よう低彩度・低不透明で「気配」に留める。

**⑥ 人の気配（強化・最重要＝3対3を絵で伝える）** — コントラストと存在感を上げる:
- **塗りを濃く**: 現行 ink-700 `#4A433C` 62〜82% → **手前の人は ink-900 `#2B2622` 88%、中列 ink-700 `#4A433C` 80%、奥列 ink-700 70%**。手前を最も濃くして「逆光シルエット」の階調を作る（一律ではなく前後で濃淡＝奥行き）。
- **サイズと配置**: 現行の3+3・向かい合いは維持。**手前の2人（左右の最前）を一回り大きく**（頭 r=11 → r=12、肩の裾を少し広げる）して主題を強調。
- **足元を地面に接地**: 現行は宙に浮き気味。前景の敷石（⑤）の上に立つよう **y位置を 188〜198 から微調整して足元を丘のラインに重ねる**（シルエットの裾が手前丘のラインに少し沈む＝接地感）。
- **顔・表情・服の柄は描かない**（現行維持。描くとクリップアート化）。逆光シルエットを貫く。
- **わずかな間隔・身長差**: 6人を機械的等間隔にせず、各人の x を ±4〜8、頭の高さを ±3 ばらつかせる（現行も多少あるが、もう少し自然な「人の集まり」に）。

### 6.3 控えめなCSS演出（補助・最小限）

SVG を引き立てる範囲で。**やり過ぎ禁止**（迷ったら入れない）。

- **額装ヴィネット（任意・推奨）**: Hero 絵の枠（LP の `rounded-lg overflow-hidden` ラッパ）に、ごく薄い内側の暗まり（vignette）を重ね、絵が締まって見えるように:
  ```
  ::after { box-shadow: inset 0 0 40px rgba(43,38,34,0.06); border-radius: inherit; pointer-events:none; }
  ```
  - rgba は ink-900 `#2B2622` の 6%。**これ以上濃くしない**（暗い額縁はSaaS臭）。
- **微グレイン（任意・入れなくてよい）**: 紙の質感を足すなら、`mix-blend-mode: multiply; opacity: 0.04` 程度の極薄ノイズSVGをHero枠に重ねる。**効果が分かりにくく管理コスト増なので、基本は入れない**。入れるなら opacity 0.03〜0.05 厳守。
- **背景アトモスフィアとの関係**: LP ヒーローの `hero-atmosphere`（globals.css）は**維持**。ただし HeroScene を強化すると、background の atmosphere と Hero 絵の空グラデが**二重で温度過多**になる懸念。→ **md+ の2カラムでは、Hero 絵は右カラムに収まり、atmosphere は section 全体に敷かれる**ので二重にならない（住み分く）。base（縦）では Hero 絵の直後に atmosphere 上のテキストが来るので、現状どおり喧嘩しない（両方とも生成り＋テラコッタ系）。**変更不要**。
- **微アニメーション**: **入れない**（落ち着き優先・`prefers-reduced-motion` 配慮）。陽がゆっくり明滅…等はSaaS臭・チラつきの元。静的を貫く。

### 6.4 「映え」の禁則（安っぽくしないための鉄則・再掲＋追加）

- 紫・青・原色・ビビッドを入れない（design-system §8）。緑は空気遠近法の淡さに留める。
- グラデは**縦/放射の2〜3色**まで。虹色にしない。
- 人影は**シルエット**（顔・服・小物を描かない）。手前を濃く・奥を淡くの**階調**で奥行きを出す（一律黒ベタにしない）。
- 光・影・前景は**ハードエッジ禁止**（radialGradient / 低不透明で溶かす）。
- フラットSaaSイラスト（極太アウトライン＋ベタ塗り＋丸目キャラ）／3Dレンダ／クリップアートにしない。**狙いは「夕暮れの逆光シルエットの1枚絵＝写真の代替」**。
- 額装の影・グレインは**極薄**（vignette 6%・grain 4%以下）。濃くすると一気に安っぽくなる。

### 6.5 将来 raster へ移行する逃げ道（今は実装しない・構造だけ）

- LP の Hero 枠（`rounded-lg overflow-hidden` ラッパ）の中身を **`HeroScene` 単体**にしておけば、将来 `<picture>`/`<img>`（自前撮影 or 許諾済み写真）へ**そのまま差し替え**できる。本書では **SVG実装を正**とする。

### 6.6 「映え」合否基準（レビュー用・更新版）

- [ ] LP を3秒見て「ちゃんとした／安心／おしゃれ」（DINE系）。**現行より明確にリッチ・コントラストがある**と感じる。
- [ ] 夕日（主光源）がはっきり存在し、空に階調（焼け→生成り）がある。
- [ ] 奥行きがある（遠景稜線→庭→前景敷石／低木の重なり）。平板でない。
- [ ] 人影が**3対3＝6人**で向かい合い、手前が濃く大きい（存在感がある）。
- [ ] 紫青・原色・ビビッド・フラットSaaS・クリップアート・3D・写真の借り物感が無い。
- [ ] PC（md+）で右カラムの Hero 絵が堂々と大きく、左のコピーと主従が成立。
- [ ] 全体が雑誌の特集扉の質感（design-system §0）。

## 7. その他のマーケ / デザイン観点の改善

実機全体を見ての気づき（#9 プロ観点）。**いずれも任意・優先度中以下**。最重要は §2〜§6。

### 7.1 LP の社会的証明・信頼の補強（マーケ）

- 現状 LP は不安解消4カードが強い。さらに信頼を上げるなら、**「運営・安全への姿勢」を1行**（例: 「身分証は確認後に削除します」「通報・退場の仕組みがあります」）を**開催についてブロックの近くに静かに**置くと、慎重な層に効く。**煽らない事実**として（design-system §0）。**任意**。
- 「数字の社会的証明」（会員数・成立数）は、**実数が伴わないなら出さない**（誇張＝信頼毀損）。出すなら実データで。当面は出さない方針が安全。

### 7.2 空状態のビジュアル（s11-polish §6.2 で設計済の再掲・任意）

- EmptyState の glyph が `◇`（文字）で寂しい。**BrandMotif（`leaf` or `lantern`、`text-ink-300` 48px）に差し替え**ると編集的に。s11-polish §6.2 で「glyph を ReactNode 化」を設計済。**未実装なら任意で実施**。色は付けない（ink-300）。PC で広い画面の中央に置かれるので、寂しさが目立ちやすい＝効果あり。

### 7.3 カードの質感（デザイン・任意）

- 一覧カードは現状 `shadow-none`（border のみ）で編集的。PC でグリッド化すると**カードが並ぶ**ので、現状の「border のみ」で十分上質（影を足すとSaaS臭）。**変更不要**。ホバーの `hover:bg-bg-sunken/60` は PC でマウス操作の手応えになり good。
- ただし PC グリッドで**カード高さがばらつく**と雑に見える → §4.1 の `h-full`/`auto-rows-fr` で揃えるのを推奨（再掲）。

### 7.4 PC で「中央カラム＋広い余白」が寂しい画面の背景処理（デザイン・任意）

- 詳細・フォーム・オンボは 480〜560px 維持（§2.5）。PC では左右に余白が出る。**寂しさ対策**として、`body`（or app-shell）の背景に**ごく淡い hero-atmosphere 相当のにじみ**を敷くと、余白が「生成りの紙の上に置かれた」見え方になり間が持つ:
  - 例: `body` に `background: radial-gradient(140% 60% at 50% 0%, rgba(194,112,61,0.05) 0%, rgba(251,247,240,0) 60%) , #FBF7F0;`（accent-500 5% の極淡い上部のにじみ）。**md 未満では出さない**（モバイルは現状の無地維持）か、出しても極淡なので影響軽微。**任意・最小限**。新トークン不要（既存透過）。
  - これにより「PC で中央カラムだけ・周りが真っ白」の寂しさが緩和。**やり過ぎ禁止**（5%厳守・紫青禁止）。

### 7.5 ロックアップ／ヘッダの PC 余白（デザイン・任意）

- AppHeader は `sticky top-0` で全幅。一覧・LP を広げると、ヘッダのタイトルが左端に貼り付く。**ヘッダ内のタイトルを本文と同じ最大幅で中央寄せ**にすると整う（`header` 内側を `mx-auto max-w-3xl lg:max-w-5xl`）。**任意**。ただしヘッダは全幅で左寄せのままでも許容範囲（多くのアプリがそう）。優先度低。

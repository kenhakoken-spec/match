## 1. 全体方針 — S11 で何を足すか（既存を壊さない差分）

S11 は **仕上げ**である。S9/S10 で確定したものは維持し、その上に「日付主役・カレンダー・映え」を上書きする差分のみを定義する。

**維持するもの（壊さない）**:
- ブランド HAKO-NIWA／箱庭、タグライン「みんなが出会える場所」（S9）。
- 料金出し分け（女性に¥2,000を見せない／公開は中立併記「男性¥2,000・女性無料」）（S9 §5）。
- garden-plot は撤去済・使わない。ヒーローは CSSアトモスフィア（S10 §2）。固定フッタCTAに戻さない（S10 §5）。
- トーン: 生成りオフホワイト＋テラコッタ1アクセント＋深緑差し色、明朝見出し×ゴシック本文、影は最小（design-system §0）。
- ボタンの押せる質感（影＋active:scale）はフェーズ1で実装済（`Button.tsx`）。本書では変えない。

**S11 で足すもの（本書）**:
- 会カードを**日付主役**に情報反転（#2）。SlotCard / PublicSlotCard / 詳細ヘッダの3か所。
- **リスト／カレンダー トグル**と月カレンダービュー（#3）。browse と explore の両方。
- LPヒーローに**主役級ビジュアル**（作り込んだSVGシーン＝箱庭・出会いの情景）＋各所の適度な映え（#8）。
- それらの**一貫性ガイド**（#5）。

## 2. #2 会カードの情報設計を「日付主役」に反転

### 2.1 現状の問題（実機で確認・コードの実体）

現行 `SlotCard.tsx` / `PublicSlotCard.tsx` の上段は:

```
[ font-serif 17px ink-900  エリア ]      [ 13px ink-500  6/13(金) 19:30〜 ]
        ↑ 視覚的主役（大・明朝・濃い）            ↑ 従（小・淡い・右肩）
```

→ **エリア（恵比寿）が主役、日付が従**。殿のFB「いつ開催かが一番大事」と逆。
詳細ヘッダ（`slots/[id]` / `explore/[id]`）も `h1=「{エリア}エリア」`（serif 22）、日付は `15px ink-700` で従。**3か所すべて逆**。

### 2.2 情報階層の原則（S11確定・全カード共通）

会選びの意思決定順は **「① いつ → ② どこ → ③ 入れるか（充足）→ ④ 条件・料金」**。視覚の重みもこの順にする。

| 優先 | 情報 | 視覚的扱い | 根拠 |
|------|------|-----------|------|
| **1（主役）** | **日・曜日・時刻** | 最大・最上・最も濃い。曜日は色で区別。 | 殿FB「いつが一番大事」。会は日時で選ぶ。 |
| 2 | エリア（恵比寿/銀座/池袋） | サブラベル（チップ or 小見出し）。日付の直下 or 右に従える。 | 場所は次点。3エリアのみで選択肢が狭い。 |
| 3 | 充足（●○＋残数） | 中段。色のみに頼らずドット形状＋テキスト（既存 FillDots 維持）。 | 「入れるか」の判断材料。 |
| 4 | 条件チップ・料金 | 下段。事実情報として静かに（既存維持）。 | 最後に確認する制約。 |
| 補 | ステータス／不可理由 | 左肩バッジ or 下段に muted（既存ロジック維持）。 | 状態は色＋ラベル＋形状（design-system §5）。 |

**日付の視覚言語（重要・全カード統一）**:
- 「日」を**大きな数字**（serif・tabular）で主役化。月は小さく前置。例: 小さく `6/` ＋大きく `13`。
- **曜日は色で区別**して可読性と「映え」を両立（DINEのカレンダー的洗練）:
  - 平日（月〜金）= `ink-700`（落ち着いた本文色）。
  - **土曜 = `state-info` `#5B7186`（くすんだ青グレー）**。
  - **日曜・祝日 = `accent-600` `#A85638`（テラコッタ濃いめ）**。
  - → 原色の赤青は使わない（design-system §8）。低彩度のトークン内で「土＝青み／日＝暖色」を表現。
- 時刻は曜日の隣に `tabular-nums` で `19:30〜`。

> なぜ「日」を数字で大きくするか: 「6/13(金) 19:30」を全部同サイズで太くすると情報の塊で読みにくい。**日付を数字主体のブロック**にすると一目で「13日・金・夜」が入り、カレンダー（#3）とも視覚言語が揃う。

### 2.3 ワイヤー（before / after）— SlotCard / PublicSlotCard 共通

**Before（現行）**:
```
┌──────────────────────────────────────┐
│ 恵比寿(serif17)            6/13(金) 19:30〜(13px灰) │  ← エリア主役・日付従(逆)
│ 女性 ●●○ (あと1)   男性 ●○○ (あと2)              │
│ あと 女性1・男性2                                │
│ ──────────────────────────────────── │
│ 〔20代限定〕              男性 ¥2,000 ・ 女性 無料 │
└──────────────────────────────────────┘
```

**After（S11・日付主役）**:
```
┌──────────────────────────────────────┐
│ ┌────┐                                 │
│ │ 6/ │ 金 19:30〜          〔恵比寿〕  │  ← 左: 日付ブロック(主役) / 右: エリアチップ(従)
│ │ 13 │ ↑曜日は色(土=青/日=暖)           │
│ └────┘ ↑大きな数字(serif)              │
│                                         │
│ 女性 ●●○ あと1   男性 ●○○ あと2         │  ← 充足(既存 FillDots 維持)
│                                         │
│ ──────────────────────────────────── │
│ 〔20代限定〕            男性 ¥2,000 ・ 女性無料 │  ← 条件チップ＋料金(既存維持・出し分け維持)
└──────────────────────────────────────┘
```

**レイアウト指定（どの情報をどの大きさ／位置に）**:

上段（日付主役ブロック）— `flex items-start justify-between gap-3`:
- 左 = **日付ブロック**（`flex items-baseline gap-1.5` ＋ その後ろに曜日・時刻）:
  - 月: `font-serif text-[14px] text-ink-500`（`6/` のように `/` まで・前置・小さく）。
  - **日**: `font-serif text-[28px] leading-none tabular-nums text-ink-900`（主役・大）。
  - 曜日: `font-sans text-[14px] font-semibold` ＋**曜日色**（§2.2）。日の右に `ml-2`。
  - 時刻: `font-sans text-[13px] tabular-nums text-ink-700`（曜日の右 or 下に `19:30〜`）。
- 右 = **エリアチップ**（従）: `ConditionChip` 流用 or 同等の pill（`bg-bg-sunken` / `border-line-200` / `text-ink-700` / `text-[12px]`）。`shrink-0`。テキストは `恵比寿`（「エリア」語は不要）。

中段（充足）: 既存 `<FillDots variant="inline" />` を `mt-3` で。**変更なし**。

下段（条件＋料金）: 既存の `border-t border-line-100 pt-3` ＋ `SlotConditionChips` ＋ 料金出し分け。**変更なし**（料金の女性非表示・中立併記は死守）。

ineligible（条件不足）時: 既存どおりカード淡色＋破線＋下に muted 理由。**日付ブロックも淡色化**（日を `ink-700`、曜日色は維持してよいが彩度を上げない）。**赤にしない**（design-system §8）。

**実装メモ（frontend）**:
- 日付分解は新ヘルパーを `datetime.ts` に追加（§6.1）。`formatDateShort` は据え置き（aria-label 等で再利用）。
- aria-label は現行のまま（「恵比寿 6/13(金) 19:30 の会の詳細」）でよい。視覚の並びが変わっても読み上げ順は意味が通る。
- カード全体リンク・testid（`slot-card-link` / `public-slot-card`）・hover挙動は**変更なし**。

### 2.4 詳細ヘッダ（`slots/[id]` / `explore/[id]`）の日付主役化

現行ヘッダ:
```
{エリア}エリア   (serif 22 / 主役)
6/13(金) 19:30〜  (15px ink-700 / 従)
3 対 3（男女各3名）
〔条件チップ〕
```

S11 ヘッダ（日付主役・編集的）:
```
┌──────────────────────────────────────┐
│  6月13日(金)                            │  ← 主役: 明朝 display 28px / 曜日色
│  19:30〜                                │  ← 時刻: sans 17px tabular / ink-700
│  〔恵比寿〕  3対3（男女各3名）            │  ← エリアチップ(従) + 人数 caption
│  〔20代限定〕                            │  ← 条件チップ(既存)
└──────────────────────────────────────┘
```

指定:
- 主役 = **`font-serif text-[28px] leading-tight text-ink-900`** で「6月13日(金)」。曜日「(金)」部分のみ曜日色（§2.2）を付ける（`<span>` で囲う）。一覧カードは「13」を数字主役にしたが、詳細は1件なので**日付フル表記＋明朝**で編集感を出す（カレンダーの「日タップで来た」文脈とも整合）。
- 時刻 = `font-sans text-[17px] tabular-nums text-ink-700` で `19:30〜`、主役の下 `mt-1`。
- エリア = チップ（`ConditionChip` 同等）として時刻の下に。人数「3対3（男女各3名）」を同じ行に `caption` で従える（`flex items-center gap-2`）。
- 条件チップ = 既存 `SlotConditionChips` 維持。
- 以降（募集状況・参加条件・料金・当日の流れ・応募フッタ）は**全て現行維持**。

> 「{エリア}エリア」h1 は廃止。エリアはチップに格下げ。**日付がページの第一声**になる。

## 3. #3 カレンダー表示（リスト／カレンダー トグル）

### 3.1 方針（最小構成・過剰実装しない）

殿FB「会を日付で見られる」。**月カレンダー1種のみ**（週ビューは作らない＝過剰回避）。リストとカレンダーを**トグル**で切替。スマホ縦で見やすさ最優先。

- 配置: **browse（認証）と explore（公開）の両方**に同じトグルを置く。日付主役（#2）の流れと整合。
- カレンダーは**開催日を示す“目次”**であり、会の本体表示はリスト。カレンダーで日を選ぶ→**その日の会カードがカレンダー直下に出る**（画面遷移なし・同一ページ内）。これが「最小で見やすい」構成。
- 既存のデータ取得（`fetchSlots` / `fetchPublicSlots`）はそのまま。**取得済みの slots をクライアントで日付グルーピングするだけ**（新API不要）。

### 3.2 トグルの位置と見た目

リスト見出し直下（browse ならステータスバナーの下、explore なら説明文の下）に**セグメント切替**を置く:

```
┌─────────────────────────────┐
│  [ リスト ]  [ カレンダー ]   │  ← セグメント(2択トグル)
└─────────────────────────────┘
```

- セグメント = `bg-bg-sunken` の角丸 `radius-full` トラックに、選択中タブが `bg-bg-surface` ＋ `shadow-sm` ＋ `text-ink-900 font-semibold`、非選択 `text-ink-500`。下部タブ（accent）とは別物（過剰装飾しない・design-system §4.4 のトーン）。
- 高さ 40px、タップターゲット確保。`role="tablist"` / 各 `role="tab"` ＋ `aria-selected`。
- 既定は **リスト**（現行体験を変えない）。トグルは `useState`（ローカル）でよい。URL同期は不要（最小構成）。

### 3.3 月カレンダーのワイヤー（スマホ縦）

```
┌──────────────────────────────────────┐
│        ‹      2026年 6月      ›         │  ← 月送り(前後月。会のある月のみ動けば十分)
│  日  月  火  水  木  金  土             │  ← 曜日見出し(日=暖色/土=青系の小ラベル)
│  ─────────────────────────────────── │
│   1   2   3   4   5   6   7            │
│                       •                │  ← 開催日にドット(accent)
│   8   9  10  11  12  13  14            │
│           •           ••               │  ← 複数会は●を最大3つ or 件数
│  15  16  17  18 [19] 20  21            │  ← 今日=[枠線] / 選択日=塗り
│           •        •                   │
│  22  23  24  25  26  27  28            │
│  29  30                                │
│  ─────────────────────────────────── │
│  6/19(木) の会                          │  ← 選択日の見出し(曜日色)
│  ┌────────────────────────────────┐  │
│  │ [日付主役カード]                 │  │  ← その日の会(#2のカードを再利用)
│  └────────────────────────────────┘  │
│  ┌────────────────────────────────┐  │
│  │ [日付主役カード]                 │  │
│  └────────────────────────────────┘  │
└──────────────────────────────────────┘
```

**セルの仕様**:
- グリッド = `grid-cols-7`、各セルは正方比に近い（`aspect-square` or 高さ44px＋）でタップターゲット確保。日数字は `text-[14px] tabular-nums`。
- 曜日見出し: `text-[11px] text-ink-500`。**日曜=`accent-600` / 土曜=`state-info`**（§2.2と統一）。
- **開催日マーク**: 日数字の下に `accent-500` の小ドット（`h-1 w-1 rounded-full`）。
  - 1件=ドット1つ。2件以上=**ドット最大2つ＋件数は出さない**（最小構成。賑やかにしない）。または「日数字を `accent-600` ＋下に1ドット」で「会あり」を表現してもよい（どちらか一方・実装簡素を優先）。**推奨: ドット1つで「会あり」を示すだけ**（件数表示は過剰）。
- **開催の無い日**: 通常 `text-ink-500`、ドットなし、**タップ不可**（`disabled` / カーソル既定）。前月・翌月のはみ出し日は `text-ink-300` で薄く（or 空セル）。
- **今日**: 日数字を `ring-1 ring-line-200` の丸 or `font-semibold text-accent-600`（控えめに。塗りつぶさない）。
- **選択日**: 日数字を `bg-accent-500 text-white rounded-full`（塗り）。今日かつ選択中なら塗りを優先。
- セルは会のある日だけ `<button>`。`aria-label="6月19日(木) 会2件"`（件数は読み上げのみ・視覚は出さない）。

**選択日の会一覧（カレンダー直下）**:
- 見出し: `6/19(木) の会`（`font-serif text-[18px]`、曜日色）。
- その日の会を **#2 の日付主役カードで再利用**して縦に積む（同じカード・同じ料金出し分け）。カード上段の日付ブロックは冗長だが**そのまま再利用でよい**（実装簡素・一貫性優先。日付は文脈で分かっていても害はない）。
- **初期選択**: カレンダーに切替えた直後は「**開催日のうち最も近い未来の日**」を自動選択して、その日の会を出す（空の状態で止めない）。今日に会があれば今日。

### 3.4 状態（空の日／今日／選択日／会ゼロ月）

| 状態 | 表示 |
|------|------|
| 開催がある日 | 日数字＋accentドット、タップ可、`<button>` |
| 開催が無い日 | 日数字（ink-500）、ドットなし、タップ不可 |
| 前月/翌月のはみ出し日 | ink-300 で薄く or 空セル（どちらでも可・空セル推奨で簡素） |
| 今日 | 日数字を控えめ強調（ring or accent-600 文字）。塗らない |
| 選択日 | 日数字を `bg-accent-500 text-white` の丸塗り |
| 選択日に会あり | 直下に日付主役カードを縦積み |
| 選択日に会なし | （上記の自動選択で基本起きないが）直下に小さく「この日に会はありません」＋「近い日: 6/19(木)」リンク。EmptyState のフル画面は使わない（カレンダーは残す） |
| その月に会が1件もない | カレンダーは出し、下に `EmptyState`（既存・glyph ◇「この月に会はありません」body「翌月以降に順次公開します」）。月送りで他月へ |
| 取得ゼロ（全体で会なし） | 現行 EmptyState を維持（トグル自体を出さない or リスト側と同じ空表示） |

### 3.5 リスト↔カレンダーの行き来

- トグルで相互に切替（同一ページ内・再取得なし）。
- カレンダーの会カードをタップ → 既存どおり詳細（`/slots/[id]` or `/explore/[id]`）へ遷移（カードは #2 のものをそのまま使うのでリンク挙動は同じ）。
- 詳細から戻るとトグル状態は**リセットされてよい**（最小構成。状態保持は不要）。気になるなら `useState` 初期値をカレンダーにしてもよいが、**既定リストで十分**。

### 3.6 カレンダーを別ファイルに切り出す（実装の置き場）

- 新規 `src/components/slots/SlotCalendar.tsx`（presentational に近い・slots 配列を受け取り、選択日 state を内部に持つ）。
- browse / explore からは「リスト or `<SlotCalendar slots={...} renderCard={...} />`」をトグルで出し分け。`renderCard` で SlotCard / PublicSlotCard を差し込めば**1つのカレンダーを両画面で共有**できる（DRY・一貫性）。
- 日付グルーピングは `datetime.ts` の新ヘルパー（§6.1）で「JSTの年月日キー」を作って `Map<string, Slot[]>` に。**SSR/CSR一致のため JST 固定オフセット**（既存 `jstParts` と同じ流儀）。

## 4. #8 映えるビジュアル（最重要・LPヒーロー＋各所）

殿の最重要要望: **「もっと画像を入れて映えさせる。TOP(LP)にちゃんと画像を」**。
現状 LP はCSSアトモスフィア（グラデ）はあるが**主役級の絵がなく平板**。

### 4.1 手段の比較と推奨

| 手段 | 内容 | 利点 | 欠点 | 判定 |
|------|------|------|------|------|
| (a) **作り込んだSVGシーン** | 箱庭／出会いの情景を抽象化。複数レイヤー・グラデfill・有機シェイプ。frontend が起こす1枚絵。 | ライセンス無リスク・currentColorとトークンで完全統制・配信失敗ゼロ（インライン）・軽量・将来差替容易。S9以来の「画像欠落で枠だけ」事故を構造的に回避。 | 作り込みに設計指定が要る（本書で出す）。 | **★推奨** |
| (b) CSSのみ幾何/有機 | div＋グラデ＋clip-pathで図形。 | 実装が一部簡単。 | 複雑な情景は破綻しやすい・「図形を置いた」感が出やすい（garden-plotの二の舞リスク）・表現の上限が低い。 | 補助のみ（背景アトモスフィアは既にCSS） |
| (c) フリー写真（unsplash等） | 実写を配置。 | 一番「映える」可能性。 | **ライセンス継続管理・人物の肖像/民族/年齢が世界観とズレるリスク・読込/CLS・加工統一が難しい・"借り物"感**。design-system §4.6「自然光・等身大・加工しすぎない」を満たす無料人物写真は実務上困難。 | **不採用（当面）**。将来、自前撮影 or 許諾済み写真が用意できたら hero に重ねられる構造（§4.3）にしておく |

**推奨 = (a) 作り込んだSVGシーン**。理由:
1. **ライセンス／品質リスクがゼロ**（殿の懸念点に直答）。
2. HAKO-NIWA は既にインラインSVG方針（BrandMotif）＝**一貫性**が取れ、currentColor/accent でトークン統制できる。
3. アトモスフィア（グラデ＝空気）＋**SVGシーン（主役の絵）**の二層で、「写真っぽい作り込み」のリッチさを外部依存なく出せる。
4. garden-plot の失敗＝「俯瞰の製図で解読を強いた」。今回は**解読不要の“情景”**（庭・灯り・人の気配）を**塗り＋グラデ＋やわらかい曲線**で描く＝雑誌の扉絵に近い。

> 「(a)が garden-plot と何が違うのか」: garden-plot は**線だけの俯瞰図解**。今回のシーンは**塗りとグラデで奥行きのある“絵”**（空のグラデ／庭／灯り／人影のシルエット）。線画アイコンではなく**1枚のイラストレーション**として作る。安っぽさの原因（線だけ・記号的・解読要求）を全て外す。

### 4.2 推奨ヒーロービジュアルの具体仕様（frontend が起こすSVG）

**名称**: `HeroScene`（新規 `src/components/brand/HeroScene.tsx`。BrandMotif の小モチーフとは別格の「主役絵」なので別ファイル）。

**構図（箱庭＝小さな庭で人が出会う夕暮れの情景）**:
- viewBox `0 0 360 280`（横長・LP幅に馴染む。レスポンシブに `width:100%`）。
- **奥行き4レイヤー**（後ろ→前）。「写真っぽい奥行き」は**重なりとグラデ**で作る:
  1. **空（背景グラデ）**: 上から下へ、`accent-100 #F6E7DC` → `bg-base #FBF7F0` の縦 linearGradient。夕暮れのあたたかい空気。彩度は上げない。
  2. **遠景の弧（陽の気配）**: 画面上部にやわらかい円弧 or 半円を `accent-300 #E7B79A` 30〜40%不透明で。沈む陽／あたたかい光源。`blur` 風に `opacity` グラデで縁を溶かす（SVG `radialGradient` fill）。
  3. **庭（中景）**: なだらかな丘の曲線を2枚、奥=`secondary-100 #E7EDE6`、手前=`secondary-500 #5E7A57` 18〜25%不透明（低彩度の緑・庭の地面）。曲線は**不均一なベジェ**（製図に見せない）。中景に**石灯籠**（既存 `lantern` モチーフの線を流用／`trust-600` 寄りの線）を1つ小さく置き「灯り＝安心」を据える。灯りの芯に `accent-500` の小さな光点＋ごく淡い放射。
  4. **人の気配（前景・主題）**: **6人ぶんのシルエット**を抽象化（顔の描き込みはしない＝個人を特定しない・等身大の気配）。丸い頭＋肩のなだらかな曲線の**塗りシルエット**を、3人＋3人でゆるく向かい合う配置。塗りは `ink-700 #4A433C` を **70〜85%不透明**（真っ黒にしない・夕暮れの逆光シルエット）。**3対3＝6人**というサービスの核を、絵で直接伝える（最重要）。

**色（全てトークン由来・新色なし）**:
- 空: `#F6E7DC`→`#FBF7F0`（accent-100→bg-base）。
- 光: `#E7B79A`（accent-300）＋芯に `#C2703D`（accent-500）。
- 庭: `#E7EDE6`（secondary-100）／`#5E7A57`（secondary-500・低不透明）。
- 灯籠の線: `#8A6D3B`（trust-600）or `#4A433C`（ink-700）。
- 人影: `#4A433C`（ink-700）70〜85%。
- **紫・青・原色は一切なし**（design-system §8）。緑は低彩度・低不透明で「庭の地面」に留める。

**トーン規定（安っぽくしないための禁則）**:
- グラデは**2〜3色の縦/放射**まで。虹色・ビビッドにしない。
- 人影は**シルエット**（顔・表情・服の柄を描かない）。描き込むと安っぽいクリップアート化する。
- 影・光は**ハードエッジ禁止**（radialGradient で縁を溶かす）。
- フラットSaaSイラスト（極太アウトライン＋ベタ塗り＋丸目のキャラ）にしない。**「夕暮れの逆光シルエット」=編集的な写真の代替**を狙う。
- アニメーション無し（落ち着き優先・`prefers-reduced-motion` 配慮も自動で満たす）。

**LP内の置き方（情報設計）**:
```
┌─────────────────────────────────┐
│  [mark] HAKO-NIWA               │  ロックアップ(現行維持)
│  箱庭 ・ 東京 恵比寿/池袋/銀座    │
│                                 │
│ ╭───────── ヒーロー ──────────╮ │  ← relative isolate
│ │ ░░░ hero-atmosphere(背後) ░░ │ │  ← 既存グラデ(維持)
│ │                             │ │
│ │  ┌───────────────────────┐  │ │
│ │  │   ▓▓ HeroScene ▓▓     │  │ │  ← ★新規: 主役SVG絵(空/庭/灯/6人影)
│ │  │  (空グラデ・灯り・6人影) │  │ │     角丸 radius-lg / 高さ 約200〜220px
│ │  └───────────────────────┘  │ │
│ │                             │ │
│ │  みんなが出会える場所         │ │  タグライン(現行)
│ │  3対3で、会いにいく。         │ │  主見出し(serif32・現行)
│ │  ┌─────────────────────┐    │ │
│ │  │ 男女3人ずつ、計6人で… │    │ │  サブ見出し白面カード(現行)
│ │  └─────────────────────┘    │ │
│ │  [   LINEではじめる   ]      │ │  主CTA(現行)
│ │  (   会を見てみる    )       │ │  副CTA(現行)
│ │  スマホのLINEで開くと…       │ │  LINE外案内(現行)
│ ╰─────────────────────────────╯ │
└─────────────────────────────────┘
```

- **HeroScene はロックアップとタグラインの間**（ファーストビューで最初に目に入る絵）。`mt-8` 程度。
- 枠: `rounded-lg overflow-hidden`（radius-lg 14px）。**枠線は付けない**（絵が額装され過ぎないよう・空グラデが地に溶ける）。影も付けない（アトモスフィアと喧嘩させない）。
- 高さ: viewBox 比で幅100%なら自然に約 200〜220px（LP幅360〜420想定）。スマホで主役だが画面を占有しすぎない。
- `aria-hidden`（装飾）。意味はタグライン・サブ見出しのテキストが担う。
- **アトモスフィア（背後グラデ）は維持**。HeroScene の空グラデと二重で温度が増す（喧嘩しない＝両方とも生成り＋テラコッタ系）。

> これで殿の「TOPにちゃんと画像を／映えない」に直答する。**写真ではないが“1枚絵”として主役級**、かつ 3対3＝6人 という価値を絵で伝える。ライセンス・配信リスクはゼロ。

### 4.3 将来 写真を入れる場合の逃げ道（今は実装しない・構造だけ）

- HeroScene を `<picture>` でラップできる構造にしておけば、将来「自前撮影 or 許諾済み写真」が用意できた時に `HeroScene` を `<img>` に差し替えるだけで済む。本書では**SVG実装を正**とし、写真は採用しない（§4.1 (c) 不採用理由）。

| 素材 | 役割 | 使う場所 |

- 緑（secondary）は HeroScene の庭・成立系のみ（低彩度）。

### 5.5 一貫性の最終チェック
- [ ] どの画面でも「日付＝主役・数字＆明朝・曜日色」「エリア＝チップで従」。
- [ ] 曜日色（平日/土/日）が全画面で一致。
- [ ] HeroScene は LP の1か所だけ。他画面に持ち込んでいない。
- [ ] 新色を足していない（既存トークンのみ）。accent を撒きすぎていない。
- [ ] 線画モチーフと塗り絵が同一視界で競合していない。

## 6. 新トークン／ユーティリティ（最小限・既存と矛盾しない）

**新カラートークンは追加しない**（design-system §7 の既存で完結）。曜日色も既存トークンの流用。
追加するのは**日付分解ヘルパー**と**EmptyState の glyph 拡張**のみ。

### 6.1 日付ヘルパー（`src/app/_lib/datetime.ts` に追記）
既存 `formatDateShort` / `formatTime` / `startMillis` は**据え置き**（aria-label・既存箇所で継続使用）。以下を**追加**（既存 `jstParts` を再利用＝SSR/CSR一致）:

```
// 日付主役カード・カレンダー用に JST の構成要素を返す。
// 返り値: { month:6, day:13, weekday:"金", weekdayIndex:5, time:"19:30", ymdKey:"2026-06-13" }
//   weekdayIndex 0=日…6=土（曜日色の判定に使う）/ ymdKey は日付グルーピングのキー。
export function jstDateParts(iso: string): {
  month: number; day: number; weekday: string; weekdayIndex: number;
  time: string; ymdKey: string; year: number;
}
```
- 実装は既存 `jstParts` を使い、`ymdKey = \`${y}-${pad2(mo)}-${pad2(d)}\``。
- 曜日色の判定ヘルパーも併設すると DRY:
```
// 曜日色クラス（§2.2）。0=日→accent-600 / 6=土→state-info / 他→ink-700。
export function weekdayColorClass(weekdayIndex: number): string
//  返り値例: "text-accent-600" | "text-state-info" | "text-ink-700"
```
- カレンダーの月グリッド生成（先頭の空セル・月内日数・前後はみ出し）は `SlotCalendar.tsx` 内のユーティリティでよい（datetime.ts に月情報ヘルパー `monthGrid(year, month)` を置くのも可）。

### 6.2 EmptyState の glyph 拡張（`src/components/States.tsx`）
- 現行 `glyph?: string`（既定 `◇`）を `glyph?: ReactNode` に広げ、**BrandMotif を渡せる**ように（§4.4）。
- 既存呼び出し（文字 `◇` 渡し）は後方互換。色は `text-ink-300`（変えない）。
- これは任意の映え向上。実装簡素を優先するなら**現行 ◇ のままでも可**。

### 6.3 トグル／カレンダーのスタイル
- 既存 Tailwind トークン（`bg-bg-sunken` / `bg-bg-surface` / `shadow-sm` / `radius-full` / `text-ink-*` / `accent-*`）で実装可能。**新ユーティリティ不要**。
- `tailwind.config.ts` は**変更不要**。
